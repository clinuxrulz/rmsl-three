# Chrome Bug: WebGPURenderer WebGL fallback with specific GLSL shaders

## The Bug

In Chrome, when `WebGPURenderer` uses the WebGL fallback (either because WebGPU init fails or via `forceWebGL: true`), the specific GLSL shader code generated by the RMSL compiler triggers a rendering failure. The same code works in Firefox.

The three.js code path: `WebGPURenderer` → `WebGLBackend` → `NodeMaterial.build()` detects `isWebGLBackend` → selects `glslFn()` with the RMSL-compiled GLSL string.

## How to reproduce

Open `index.html` in **Chrome** and **Firefox** side by side.

- **Firefox**: renders the animated sphere correctly
- **Chrome**: fails (blank canvas, console errors, or crash)

## What's included

- **Zero npm dependencies** — three.js loaded from CDN via importmap
- **Pre-compiled shader strings** — GLSL and WGSL extracted from the RMSL compiler, embedded as constants
- **Exact same three.js API flow** as the original demo: `WebGPURenderer` → `NodeMaterial.build()` → `glslFn()`/`wgslFn()` → `super.build()`
